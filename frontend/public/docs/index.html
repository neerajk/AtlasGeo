Atlas GeoAI Documentation

Atlas GeoAI is a natural-language satellite imagery search and analysis platform. Describe what you need in plain English — Atlas searches the Sentinel-2 archive, runs geospatial analysis, and renders results on an interactive globe.

💬
Natural Language Search
Find satellite scenes by describing place, date, and cloud cover in plain text.
🔬
Analysis Tools
NDVI, NDWI, NDBI, flood mapping, burn scar detection — click to run.
🎯
Scene Picker
Browse available scenes on the map before committing to analysis.
🗂️
Layer Management
Toggle, adjust opacity, and remove analysis output layers.

Getting Started

Prerequisites

  • micromamba (or conda) for the Python environment
  • Ollama installed and running locally
  • Node.js ≥ 18 for the frontend

1. Clone & create environment

git clone https://github.com/yourorg/atlas-geoai.git
cd atlas-geoai

micromamba create -f environment.yml
micromamba activate atlas

2. Pull the LLM model

ollama pull gemma4:31b-cloud
Note
The default model is gemma4:31b-cloud. You can change it in config/models.yaml. Any Ollama model works — smaller models (e.g. llama3.2:3b) are faster but less accurate at extracting search parameters.

3. Configure environment

cp .env.example .env
# Edit .env — only CORS_ORIGINS is required for local dev

4. Start all services

# Terminal 1 — Backend
uvicorn atlas.main:app --reload --port 8000

# Terminal 2 — Tile server
uvicorn titiler.application:app --port 8001

# Terminal 3 — Frontend
cd frontend && npm install && npm run dev

Open http://localhost:5173 — you should see the globe with the chat panel on the right.

Your First Query

Type in the chat box and press Enter:

Show Sentinel-2 images over Nairobi, Kenya from last month

Atlas will search the archive, display footprints on the globe as blue polygons, and return a summary table of scenes in the chat.

Natural Language Search

Atlas uses an LLM to extract structured search parameters from your query. You don't need to remember API syntax — describe what you want naturally.

Query structure

A good query has three parts — location, date, and optionally cloud cover. Analysis type is detected automatically from keywords:

PartExamples
LocationMumbai, India · Amazon rainforest · latitude/longitude bbox · Bangladesh
Datelast month · March 2024 · 2024-01-01 to 2024-03-31 · this week · latest
Cloud coverless than 5% cloud · clear scenes · any cloud cover
TaskNDVI · flood extent · burn scar · vegetation health · urban extent

Example queries

Show Sentinel-2 images over Nairobi, Kenya from last month
NDVI analysis over the Amazon rainforest in January 2024 with less than 10% cloud
Flood extent in Bangladesh after July 2024 monsoon
Burn scar mapping over California, October 2023
NDBI urban extent analysis over Mumbai, March 2024
Latest clear scenes over London

Date expressions

Relative expressions are resolved against today's date at query time:

ExpressionResolves to
last monthFull calendar month before today
this weekLast 7 days
latest / recent / newestLast 30 days
March 20242024-03-01 → 2024-03-31
2024-01-01 to 2024-03-31Exact range
last 6 months180 days back from today

Locations & bbox

Atlas resolves place names using the LLM's built-in geography knowledge, with a Nominatim geocoder fallback for unknown places. You can also provide explicit coordinates:

Sentinel-2 images over bbox [72.7, 18.8, 73.1, 19.3] (Mumbai)

Cloud cover

Default is <20% cloud cover. You can override it:

PhraseCloud cover max
clear scenes / less than 5%5%
(default)20%
up to 40% cloud40%
any cloud cover / cloudy ok80%

Automatic search fallback

If no scenes match your original criteria, Atlas automatically relaxes constraints and retries — up to 5 times:

AttemptCloud maxDate window
1 (original)As requestedAs requested
220%Same
340%Same
440%+90 days back
5 (last resort)80%+180 days back

When a fallback is used, the response notes which parameters were actually applied.

Analysis Tools

Atlas includes five analysis tools that run on Sentinel-2 COG (Cloud-Optimized GeoTIFF) bands. All outputs are float32 or uint8 GeoTIFFs rendered as map layers via titiler.

Cloud masking
All analysis tools apply the Sentinel-2 Scene Classification Layer (SCL) to mask clouds (class 8/9), cloud shadows (class 3), and thin cirrus (class 10) before computing indices. Results are more accurate than raw band math alone.
ToolTask keywordMethodOutput
NDVIndvi, vegetation, plant health(NIR − Red) / (NIR + Red)Float32 [-1, 1]
NDWIndwi, water body, river(Green − NIR) / (Green + NIR)Float32 [-1, 1]
NDBIndbi, urban extent, built-up(SWIR1 − NIR) / (SWIR1 + NIR)Float32 [-1, 1]
Flood Mappingflood, flooding, inundationMNDWI > 0 thresholdBinary uint8
Burn Scarburn, wildfire, fireNBR < 0.2 thresholdBinary uint8

Scene Picker

When you request an analysis (NDVI, flood, burn scar, etc.), Atlas first shows all matching scenes and lets you pick which one to analyze — instead of automatically using the least-cloudy scene.

How it works

  1. Submit an analysis query (e.g. "NDVI over Mumbai, March 2024")
  2. Atlas searches the archive and displays matching scene footprints on the map as orange polygons
  3. The chat panel shows "Found N scenes. Click a scene on the map to run NDVI analysis."
  4. The map enters picker mode: cursor becomes a crosshair, a banner appears at the top of the map
  5. Hover over any footprint to see a tooltip with scene ID, date, and cloud cover %
  6. Click a footprint to select it — analysis starts immediately
  7. The selected scene's analysis result is rendered as a new layer on the map
Cancelling selection
Click Cancel in the green picker bar at the bottom of the chat panel to exit picker mode without running analysis.
Connection loss during selection
If the WebSocket disconnects while you are in picker mode, picker mode is cancelled automatically with a notification. Re-run your query to try again.

NDVI — Vegetation Health

Spectral Index Float32 Output

The Normalised Difference Vegetation Index measures the density and health of green vegetation. Values range from −1 to +1; healthy dense vegetation is typically 0.4–0.9.

Formula

NDVI = (NIR − Red) / (NIR + Red)
     = (B08 − B04) / (B08 + B04)

Interpretation

NDVI rangeMeaning
< 0Water, snow, clouds, bare rock
0.0 – 0.2Bare soil, desert, urban surfaces
0.2 – 0.4Shrubland, grassland, sparse vegetation
0.4 – 0.6Moderate vegetation, cropland
0.6 – 0.9Dense healthy vegetation, tropical forest

Example queries

NDVI over the Amazon rainforest in January 2024
Vegetation health map of Punjab croplands, harvest season 2024
Plant health index over Nairobi, last month

Output details

  • Format: Float32 GeoTIFF, values clipped to [−1, 1]
  • Nodata: −9999
  • Resolution: 512 × 512 (resampled from 10m NIR + 10m Red)
  • Colormap: rdylgn (red → yellow → green)
  • Rendered via titiler with rescale=-1,1

Bands used

BandSentinel-2 asset keyNative resolution
NIRnir (B08)10m
Redred (B04)10m
Cloud maskscl20m (upsampled)

NDWI — Water Bodies

Spectral Index Float32 Output

The Normalised Difference Water Index highlights open water bodies. Positive values typically indicate water; negative values indicate vegetation or dry land.

Formula

NDWI = (Green − NIR) / (Green + NIR)
     = (B03 − B08) / (B03 + B08)

Interpretation

NDWI rangeMeaning
> 0.3Open water (lakes, rivers, flooded areas)
0.0 – 0.3Wet soil, shallow water
< 0Dry land, vegetation, urban

Example queries

Water body extent over Lake Victoria, June 2024
NDWI analysis over the Nile delta, surface water detection
River extent mapping over Brahmaputra, Bangladesh

Output details

  • Format: Float32 GeoTIFF, values clipped to [−1, 1]
  • Colormap: blues_r (white → blue, higher = more water)
  • Rendered with rescale=-1,1

NDBI — Built-up / Urban Extent

Spectral Index Float32 Output

The Normalised Difference Built-up Index identifies urban and impervious surfaces. Built-up areas have high SWIR1 reflectance relative to NIR.

Formula

NDBI = (SWIR1 − NIR) / (SWIR1 + NIR)
     = (B11 − B08) / (B11 + B08)

Interpretation

NDBI rangeMeaning
> 0.1Built-up land, concrete, asphalt
−0.1 – 0.1Bare soil, transition areas
< −0.1Vegetation (inverse relationship)

Example queries

NDBI analysis over Mumbai to show urban extent, March 2024
Urban sprawl mapping over Nairobi outskirts, 2024
Built-up index over Beijing metropolitan area, last month

Bands used

BandSentinel-2 asset keyNative resolution
SWIR1swir16 (B11)20m
NIRnir (B08)10m
Cloud maskscl20m

Flood Mapping

Binary Mask Area Stats

Flood mapping uses MNDWI (Modified Normalised Difference Water Index) to detect inundated areas. Pixels with MNDWI > 0 are classified as water/flood.

Formula

MNDWI = (Green − SWIR1) / (Green + SWIR1)
      = (B03 − B11) / (B03 + B11)

This method is used by the Copernicus Emergency Management Service (CEMS) for operational flood mapping.

Example queries

Flood extent in Bangladesh after the July 2024 monsoon
Inundated areas over Pakistan Sindh province, August 2023
Water extent mapping over the Mekong delta

Output details

  • Format: Binary uint8 GeoTIFF (0 = not flooded, 1 = flooded)
  • Rendered as a blue semi-transparent overlay (rescale=0,1)
  • Chat response includes flooded area in km² and pixel count
Limitation
MNDWI detects current surface water from a single image — it does not distinguish between permanent water bodies and flood. For change detection, compare two scenes from before and after the flood event.

Burn Scar Detection

Binary Mask Area Stats

Burn scar detection uses NBR (Normalised Burn Ratio) to identify areas affected by wildfire. Burned pixels have significantly lower NBR than healthy vegetation.

Formula

NBR = (NIR − SWIR2) / (NIR + SWIR2)
    = (B08 − B12) / (B08 + B12)

Thresholds

NBR rangeBurn severity
< −0.25High severity burn
−0.25 – 0.1Moderate burn
0.1 – 0.2Low / unburned (threshold at 0.2)
> 0.3Healthy vegetation

Example queries

Burn scar mapping over California wine country, October 2023
Wildfire damage over Australian bush, February 2024
NBR fire scar detection over Siberia, summer 2024
Single-image limitation
The current tool computes NBR on a single post-fire image. Bare soil and dark rock naturally have low NBR and may be misclassified as burned. For higher accuracy, a pre-fire reference scene should be used to compute dNBR (delta NBR). This is on the roadmap.

Chat Panel

The right-side panel is your interface to Atlas. Type a query, press Enter or click →. While Atlas is working, a step-by-step progress indicator shows the current stage.

Progress stages

  1. Planning search… — LLM extracts bbox, date range, task type
  2. Searching satellite archives… — STAC query to Element84
  3. Found N scenes. Formatting… — rendering results
  4. (For analysis tasks) Running analysis… — computing index/mask
  5. Rendering layer… — COG layer added to map

Conversation History

Atlas maintains conversation context within a session. Follow-up queries understand previous context:

Show Sentinel-2 images over Mumbai in March 2024
Now run NDBI on the clearest scene
What was the mean value?
Session scope
Conversation history is per-browser-session. Refreshing the page starts a new conversation. The history is sent with each query as a list of {role, content} pairs and prepended to the LLM's context.

Troubleshooting

No scenes found

Atlas automatically relaxes cloud cover and widens the date window (up to 5 retries). If still empty:

  • Try a broader date range: "…from the last 6 months"
  • Increase cloud cover: "…with up to 40% cloud cover"
  • Check the region has Sentinel-2 coverage — the archive starts mid-2015

Analysis output looks wrong / too dark

  • The scene may have high cloud cover despite the filter — pick a scene with lower cloud % in picker mode
  • Try a different date — seasonal vegetation changes affect NDVI significantly

Ollama not responding

ollama serve        # ensure Ollama is running
ollama list         # check model is pulled
ollama pull gemma4:31b-cloud

titiler errors in tile rendering

  • Ensure titiler is running on port 8001: uvicorn titiler.application:app --port 8001
  • Check VITE_TITILER_URL in your frontend .env matches

WebSocket keeps reconnecting

  • Check the backend is running: curl http://localhost:8000/health
  • Verify CORS_ORIGINS includes your frontend origin (e.g. http://localhost:5173)

Output GeoTIFFs deleted on restart

Output files older than 24 hours are automatically cleaned up at startup. This is by design — save any outputs you want to keep before restarting the backend.
